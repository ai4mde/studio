# Prototype Generation Roadmap

## Background

The current prototype generation flow is already able to produce functional Django prototypes from diagram and interface metadata, but its main rendering path is still centered on a business tree of `pages -> sections -> attributes/operations`. This structure is useful for functional generation, yet it is not sufficient for a future workflow where page layout needs to adapt to domain-specific archetypes, imported template references, or Figma-derived structures.

The next stage should therefore preserve the current strengths of the system while adding a stronger composition layer between business semantics and final rendering.

## Objectives

The target architecture should satisfy the following goals:

1. Keep diagram-derived business semantics as the primary source of truth.
2. Preserve current manual authoring for interfaces, sections, and frontend components.
3. Keep theme and styling as first-class metadata rather than mixing them into structural generation logic.
4. Support adaptive page structures based on interface metadata, diagram context, and reusable template references.
5. Generate multiple candidate page trees for human review instead of forcing a single deterministic structure too early.
6. Ensure preview and final prototype are driven by the same selected structural description.
7. Keep authentication pages and similar platform pages as stable system templates in the first phase.

## Core Principle

The system should distinguish clearly between three different concerns:

1. Business semantics: what a page or section needs to do.
2. Composition: where those capabilities are placed on the page and how regions/components are arranged.
3. Visual styling: how the selected structure is themed and styled.

This separation is the main condition for supporting both adaptive layouts and stable prototype generation.

## Proposed Metadata Layers

### 1. Semantic Layer

This layer remains grounded in diagrams and interface definitions. It includes:

- page intent
- section intent
- data entities and relations
- operations and actions
- user role / actor information
- workflow context

In current terms, this is mostly derived from diagrams plus the existing `pages` and `sections` metadata.

### 2. Composition Layer

This is the new central layer that should be added. It records the selected structural plan for each page, for example:

- page archetype
- layout skeleton
- region definitions
- section-to-region mapping
- component variants inside each region
- ordering, grouping, prominence, and fallback rules

This layer should become the shared source for both preview rendering and final prototype generation.

### 3. Style Layer

This layer holds visual tokens and styling choices, such as:

- color system
- typography
- density / spacing preferences
- component tone and shape
- theme variants
- higher-level styling summaries generated by the designer flow

This corresponds to the current `theme` and `styling` responsibilities and should stay separate from composition.

### 4. Session / Design History Layer

This layer records generation process state, for example:

- candidate variants
- human feedback
- selected variant
- design rationale
- provenance of imported templates
- designer session metadata

This is where the existing `designerMeta` and `designerSession` ideas naturally fit.

## Target Building Blocks

### Diagram Semantics Extractor

This stage derives normalized business capabilities from diagrams and current interface metadata. It should answer questions such as:

- Is this page a product detail page, comparison page, approval page, list page, dashboard, or wizard?
- Which sections are primary, secondary, supporting, or contextual?
- Which actions require strong prominence?
- Which data groups belong together?

The result should not decide concrete layout yet. It should only produce a normalized capability profile.

### Template Skeleton Registry

The system should support reusable structural references, potentially imported from Figma or curated manually. A template skeleton should describe:

- named regions
- allowed region capacities
- preferred component families
- responsive behavior
- content prominence rules
- optional constraints for specific page archetypes

This registry is not a final page implementation. It is a structural reference library used by the generator.

### Section Capability Profiles

Each section should be mapped into a capability profile, for example:

- `hero-summary`
- `gallery`
- `filters`
- `detail-form`
- `table-list`
- `related-items`
- `timeline`
- `approval-actions`

This profile allows the generator to place sections into compatible regions without relying only on section names.

### Variant Generator

For each page, the system should produce three composition variants. Each variant should be a full page tree, not just a theme change.

Each variant should differ in structure, for example:

- emphasis on conversion vs. exploration
- dense enterprise detail view vs. lightweight summary-first layout
- tabbed vs. split-pane vs. stacked arrangement

All three variants must still satisfy the same semantic requirements from the source diagrams and interface metadata.

### Human-in-the-Loop Review

The user should review the three generated variants, compare them, and select one as the active composition. The selected result should then be persisted back into metadata as the chosen structural plan.

This step is important because layout suitability is often domain-sensitive and cannot always be inferred reliably from diagrams alone.

### Composition Metadata Emitter

After selection, the system should save a normalized `composition` structure into interface metadata. This structure should be stable enough to drive:

- preview rendering
- functional prototype generation
- later manual refinement
- regeneration with minimal accidental structural drift

## Proposed Metadata Shape

The exact schema can evolve, but the responsibilities should be roughly split as follows:

- `pages` and `sections`: business-level intent and functional content
- `layout`: global page layout preference or chosen skeleton identifier
- `composition`: concrete selected page tree with regions and mapped components
- `theme`: visual design tokens
- `styling`: style summary, designer refinements, and editor-facing visual settings
- `designerMeta` / `designerSession`: candidate history, rationale, and review state

The important point is that `composition` becomes the bridge between business structure and final rendering.

## Preview and Prototype Consistency

The current system has a conceptual split between AST-oriented preview structures and the Jinja-based functional generation path. The future architecture should reduce that split.

The recommended approach is:

1. Generate or edit a composition tree.
2. Render preview from that composition tree.
3. Generate the final Django/Jinja prototype from that same composition tree.

This does not require preview and prototype to use the exact same renderer implementation, but both must consume the same structural source. Otherwise, the preview can drift away from the generated prototype.

## Relationship to Manual Authoring

Manual authoring should remain a first-class mode.

This means:

1. A user must be able to define interfaces and sections directly without running the automatic composition pipeline.
2. A manually authored page should still be expressible in the same metadata model, especially through `sections`, `layout`, and `composition`.
3. Automatic generation should produce editable metadata, not a locked artifact.
4. Section-level handcrafted components should be preservable and re-bindable inside a generated composition.

The architecture should therefore treat generation as an accelerator, not as the only valid authoring path.

## Authentication and Stable System Pages

Authentication pages, platform shell pages, and similar system-owned views should remain template-driven and stable in the first implementation phase.

They should not go through the full three-variant composition workflow until the business-page composition pipeline is mature. This keeps critical flows predictable and reduces migration risk.

## Figma and External Template References

Figma or similar external sources should be treated as skeleton inputs, not as direct final code generators.

Their role is to provide:

- region structures
- hierarchy clues
- responsive layout ideas
- archetype exemplars

The system should normalize those imports into internal `TemplateSkeleton` definitions so that generation stays deterministic and domain-aware. If MCP-based retrieval or import tooling is added, it should support this normalization step, not replace the internal composition model.

## Suggested First Milestone

The first milestone should focus on one realistic business case, such as a bol.com-style product detail page. That case is large enough to exercise the architecture because it includes:

- hero content
- product summary
- pricing / primary actions
- structured attributes
- related items
- review or trust signals
- navigation and supporting sections

For this milestone, the system should:

1. derive semantics from diagrams and interface metadata
2. match the page to three reusable skeletons
3. produce three composition variants
4. allow a human to select one variant
5. persist the selected `composition`
6. render both preview and Django prototype from the selected result

## Implementation Sequence

1. Define internal schemas for `TemplateSkeleton`, `SectionCapabilityProfile`, and `PageComposition`.
2. Add a semantics extraction step that maps current `sections` into capability profiles.
3. Build a small curated skeleton library for one domain case.
4. Implement three-variant composition generation for that case.
5. Persist the selected composition into interface metadata.
6. Refactor preview to consume composition.
7. Refactor prototype generation to consume the same composition while preserving current Jinja strengths.
8. Add manual override support at region and section placement level.
9. Expand to more page archetypes and more domains.

## Expected Outcome

If implemented this way, the system keeps the strengths of the current generator while gaining a more flexible structural layer. That makes it possible to support domain-specific layouts, template-driven page archetypes, human-reviewed variation, and stronger consistency between design-time preview and final generated prototype.
